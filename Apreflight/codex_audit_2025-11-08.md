## 2. Partial / Drift
- LED Blinky and Scorekeeper frontends bypass the gateway by hardcoding FastAPI URLs (`frontend/src/services/ledBlinkyClient.js:6`, `frontend/src/services/ledBlinkyClient.js:20`, `frontend/src/services/scorekeeperClient.js:6`, `frontend/src/services/scorekeeperClient.js:50`, `frontend/src/services/scorekeeperClient.js:90`), so those writes skip the `/api/local` proxy and its audit logging/x-scope enforcement.
- LightGunsPanel fetches `/api/gunner/*` directly from the browser (`frontend/src/panels/lightguns/LightGunsPanel.jsx:76`, `frontend/src/panels/lightguns/LightGunsPanel.jsx:98`, `frontend/src/panels/lightguns/LightGunsPanel.jsx:118`), but the gateway only proxies `/api/local`, `/api/console_wizard`, `/api/voice`, `/api/controller`, and `/api/launchbox` (`gateway/server.js:104`), so these calls never reach FastAPI unless the frontend talks to the backend port directly.
- LED test requests still return mock acknowledgements instead of speaking to Blinky hardware, so UI “Test” buttons can’t verify cabinet wiring (`backend/routers/led_blinky.py:76`, `backend/routers/led_blinky.py:117`).
- Scorekeeper Sam’s preset buttons merely raise “Available after A: drive migration” alerts instead of loading real rosters, so the UI can’t yet seed brackets from families/friends lists (`frontend/src/panels/scorekeeper/ScoreKeeperPanel.jsx:1092`).
- LaunchBox LoRa still pings `http://localhost:8000/health` before loading data (`frontend/src/panels/launchbox/LaunchBoxPanel.jsx:285`), which bypasses the gateway and fails whenever the backend only listens behind the proxy.

## 3. Missing / Not Implemented
- There is no `/api/gunner` proxy route, so the light-gun endpoints exposed by FastAPI (`backend/routers/gunner.py:32`) cannot be reached through the sanctioned `/api/local` surface or logged (`gateway/server.js:104`).
- Console Wizard never implemented the “Restore” half of the Preview → Apply → Restore contract—the router only exposes `/retroarch/preview` and `/retroarch/apply` (`backend/routers/console_wizard.py:11`, `backend/routers/console_wizard.py:69`), so there is no way to roll back or log restore operations for RetroArch configs.
- ScoreKeeper Sam lacks any undo/restore endpoint even though it records backup paths: the router stops at submit/tournament/report handlers (`backend/routers/scorekeeper.py:233`, `backend/routers/scorekeeper.py:541`) and never adds a `/scores/.../restore`, so backups referenced in responses cannot be applied.
- Dewey still runs entirely on mock state—profiles are hard-coded in `userData` and responses are generated by `simulateDeweyResponse` after a timeout (`frontend/src/panels/dewey/DeweyPanel.jsx:31`, `frontend/src/panels/dewey/DeweyPanel.jsx:143`), so there is no documentation seeding or real backend chat wiring.
